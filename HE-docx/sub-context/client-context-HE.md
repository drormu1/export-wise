## מטרה
להגדיר את אחריויות ה-frontend עבור חוויית AI מסייעת ברורה ואמינה.
הלקוח הוא שכבת תצוגה ואינטראקציה בלבד.

## סטאק Frontend
- Angular
- חוויית משתמש בסגנון צ'אט
- תקשורת REST עם ה-API של ה-backend

## הפרדה ארכיטקטונית (מבט צד לקוח)
הלקוח חייב לכבד גבולות backend קשיחים:
- SQL Database הוא מקור האמת העסקי (בבעלות backend).
- Vector DB הוא שכבת השליפה הסמנטית (בבעלות backend).
- LLM הוא שכבת יצירת ההמלצה המייעצת (בבעלות backend).

אפליקציית Angular לא ניגשת ישירות ל-SQL, ל-Vector DB או ל-LLM.
כל האינטראקציות עוברות דרך חוזי API.

## אחריויות הלקוח
- קבלת שאלות משתמש וקלטים מובנים לבקשות.
- שליחת בקשות ל-endpoints של ה-API.
- הצגת תשובות מייעצות מבוססות grounding.
- הצגת מקרים דומים, סיכונים, מגבלות/תנאים ורמת ביטחון.
- הצגת מצבי loading, empty-state ו-error-state בצורה ברורה.
- הצגה קבועה של טקסט הסתייגות מייעצת בתצוגת התשובה.

## מה לא באחריות הלקוח
- אין לוגיקת החלטה עסקית בצד לקוח.
- אין לוגיקת retrieval/ranking ישירה בצד לקוח.
- אין אורקסטרציית prompt בצד לקוח.
- אין הרצת העשרת נתונים או אינדוקס בצד לקוח.

## זרימת חוויית משתמש (צ'אט/חיפוש)
1. המשתמש שולח שאלה.
2. הלקוח שולח בקשה ל-API.
3. הלקוח מציג מצב טעינה.
4. ה-API מחזיר תשובה מבוססת retrieval.
5. הלקוח מציג:
   - answer
   - facts מול analysis
   - recommendation
   - confidence
   - risks
   - limitations/conditions
   - supporting similar cases
   - advisory disclaimer

## זרימת חוויית משתמש (טריגר טעינת נתונים - מסך אדמין אופציונלי)
אם הלקוח חושף פעולה פנימית/אדמינית מבוקרת להעשרה:
1. המשתמש מספק `iteration`.
2. הלקוח קורא ל-endpoint טעינת הנתונים ב-API.
3. הלקוח מציג מדדי תוצאת טעינה.
4. הלקוח מציג סטטוס אינדוקס (success/partial/failed).

הערה: פעולה זו אופציונלית להיקף ה-UI ויכולה להישאר מחוץ ל-UI ציבורי.

## חוזה תלות API (הנחת צד לקוח)
הלקוח מניח שה-API מטפל ב:
- כתיבה וקריאה ל-SQL
- אינדוקס Vector אחרי iteration
- hybrid retrieval ו-grounding
- יצירת פלט מייעץ באמצעות LLM

הלקוח צריך להתייחס לתגובות API כחוזה האינטגרציה היחיד בזמן ריצה.

## כללי תצוגה לאמינות
- הפרדה ברורה בין ראיות עובדתיות לבין ניתוח AI.
- הצפת אי-ודאות כאשר ה-API מספק אותה.
- אין להציג פלט כהחלטת אישור/דחייה רשמית.
- יש לשמור על ניסוח עקבי של "תמיכה מייעצת".
