## מטרה
מסמך זה מגדיר את תחומי האחריות, החוזים וההתנהגות התפעולית של שכבת ה-Vector DB.
הוא נועד לשמור על הפרדה ארכיטקטונית נקייה מ:
- בעלות הנתונים העסקיים בשכבת SQL Database
- שכבת הנימוק והניסוח של ה-LLM

## גבולות שכבות

### SQL Database
- מחזיק את האמת העסקית.
- מחזיק את רשומות הישויות וההחלטות הקנוניות.

הערת ספק:
- ברירת מחדל נוכחית: SQLite
- ספק אופציונלי/יעד: SQL Server

### Vector DB
- מחזיק רק ארטיפקטים של אינדקס סמנטי.
- שומר embeddings ו-retrieval metadata שנגזרים מנתוני SQL.
- אסור להתייחס אליו כאחסון עסקי קנוני.

### LLM
- צורך ראיות שנשלפו.
- מייצר פלט מייעץ.
- אסור לו לשמור או להמציא עובדות עסקיות.

## כלל מקור קלט
כל המסמכים הממווקטורים חייבים לנבוע מרשומות SQL בלבד.
אין כתיבות עסקיות ידניות ישירות ל-Vector DB מחוץ לזרימות אינדוקס מבוקרות.

## חוזה בניית מסמכים
כל מסמך מאונדקס צריך לכלול:
- זהות מורכבת: `manufacturer_code`, `product_code`, `country_code`
- שדות קריאים לשיפור רלוונטיות שליפה:
  - הקשר יצרן
  - הקשר מוצר
  - הקשר מדינה
  - סטטוס החלטה
  - מגבלות
- תגיות metadata אופציונליות לדירוג/סינון
- שדות הפניה למקור לצורך traceability לרשומות SQL

## כללי הפעלת אינדוקס
אינדוקס Vector הוא חובה אחרי כל ריצת העשרת SQL מבוססת iteration שהצליחה.

רצף נדרש:
1. זיהוי רשומות SQL חדשות או ששונו.
2. בניית מסמכי retrieval מנורמלים.
3. יצירת embeddings.
4. ביצוע upsert ל-Vector DB.
5. רישום הצלחה/כשל לכל batch.

## אידמפוטנטיות והתנהגות התנגשות
- הרצה מחדש של אותה פעולת אינדוקס חייבת להיות בטוחה.
- מפתחות מסמך כפולים חייבים להוביל להתנהגות upsert דטרמיניסטית.
- כשלים חלקיים ב-batch חייבים להיות מבודדים וברי דיווח.

## טיפול בכשלים
- אינדוקס לא יכול להיכשל בשקט.
- המערכת חייבת לחשוף:
  - סך המסמכים שנוסו
  - מסמכים שאונדקסו
  - מסמכים שנכשלו
  - פרטי כשל שניתן לבצע עליהם retry
- חייבת להיות תמיכת retry בטוח בלי לפגוע בעקביות האינדקס.

## מדיניות עדכון ומחיקה
- עדכוני SQL חייבים להשתקף לבסוף ב-Vector DB.
- מחיקות SQL צריכות לסמן/להסיר מסמכי vector תואמים דרך כלל סנכרון מבוקר.
- סטייה בין SQL ל-Vector חייבת להיות ניתנת לזיהוי.

## חוזה שליפה עבור API
שכבת השליפה ב-API צריכה להשתמש ב-Vector DB לדמיון סמנטי ולשלב הקשר SQL בעת הצורך.

פלט צפוי ל-grounding של ה-LLM בהמשך הזרימה:
- המקרים הרלוונטיים ביותר
- ציון דמיון/רלוונטיות
- זהות מקור (מפתח מורכב)
- קטעי ראיות מרכזיים

## Observability
איתותי ניטור מינימליים:
- תפוקת אינדוקס לכל ריצה
- שיעור כשלים באינדוקס
- פער אינדקס מה-iteration האחרון ב-SQL
- אינדיקטורים לאיכות פגיעות בשליפה

## הרחבה עתידית
המימוש חייב להישאר אגנוסטי לספק באמצעות abstraction ל-Vector.
ספקים אפשריים: Qdrant, PostgreSQL + pgvector, Azure AI Search, Elasticsearch ואחרים.

החלפת ספק לא אמורה לדרוש כתיבה מחדש של שכבת העסק.
